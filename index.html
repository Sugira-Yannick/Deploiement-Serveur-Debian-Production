3. Sécurisation du Service SSH

Durcissement SSH

Contrôle d'Accès Réseau ISO 27001 A.9

SSH est souvent la principale porte d'entrée vers un serveur. Son durcissement est critique pour prévenir les attaques par force brute et l'accès non autorisé. La configuration suit le principe Zero Trust : rien n'est autorisé par défaut, tout doit être explicitement configuré.

3.1 Configuration avancée de SSH
# Fichier : /etc/ssh/sshd_config

Port 2222
Protocol 2

PermitRootLogin no
PasswordAuthentication no
PubkeyAuthentication yes
PermitEmptyPasswords no

AllowUsers prenom.nom admin yannick nkuru landry

MaxAuthTries 3
LoginGraceTime 30

X11Forwarding no
AllowTcpForwarding no
AllowAgentForwarding no
PermitTunnel no

UsePAM yes
Conformité ISO 27001 A.9.2.3 : Authentification forte via clés cryptographiques, limitation des tentatives de connexion, et désactivation des fonctionnalités non essentielles.
3.2 Redémarrage du service
sudo systemctl restart ssh
3.3 Génération et déploiement des clés SSH
# Sur le client :
ssh-keygen -t ed25519 -f ~/.ssh/id_ed25519_prod

# Copie de la clé publique sur le serveur :
ssh-copy-id -p 2222 -i ~/.ssh/id_ed25519_prod.pub admin@192.168.255.74

# Connexion de test :
ssh -p 2222 -i /root/.ssh/id_ed25519_prod admin@192.168.255.74

4. Hardening Système et Noyau

Renforcement de la Sécurité Système

Défense en Profondeur NIST CSF

Le hardening système implémente le principe de défense en profondeur en ajoutant plusieurs couches de sécurité. Les configurations PAM et noyau réduisent les vecteurs d'attaque et renforcent la résilience du système contre les exploits.

4.1 Politique de mots de passe (PAM)
# Fichier : /etc/pam.d/common-password

Conformité ISO 27001 A.9.2.4 : Politique de mots de passe robuste avec longueur minimale, complexité et limitation des tentatives, répondant aux exigences de l'ANSSI et du RGPD pour la protection des comptes.
4.2 Durcissement du noyau
# Fichier : /etc/sysctl.d/99-hardening.conf

net.ipv4.conf.all.rp_filter = 1

net.ipv4.ip_forward = 0
net.ipv4.tcp_syncookies = 1
net.ipv4.conf.all.accept_redirects = 0
net.ipv6.conf.all.accept_redirects = 0
net.ipv4.conf.all.accept_source_route = 0
kernel.randomize_va_space = 2

kernel.dmesg_restrict = 1
# Application des modifications :
sudo sysctl --system

5. Pare-feu nftables

Contrôle d'Accès Réseau

Segmentation et Filtrage ISO 27001 A.13.1

Le pare-feu nftables implémente une politique deny-by-default (refus par défaut) avec des exceptions explicites. Cette approche minimise la surface d'attaque réseau et permet un contrôle granulaire des flux conformément au principe de moindre privilège.

5.1 Configuration nftables complète
# Fichier : /etc/nftables.conf

#!/usr/sbin/nft -f
flush ruleset

table inet filter {
  chain input {
    type filter hook input priority 0;
    policy drop;

    iif lo accept
    ct state established,related accept
    ip protocol icmp accept
    ip6 nexthdr icmpv6 accept
    tcp dport 2222 ct state new limit rate 10/minute accept
    limit rate 5/minute log prefix "NFT DROP: "
    drop
  }

  chain forward {
    type filter hook forward priority 0;
    policy drop;
  }

  chain output {
    type filter hook output priority 0;
    policy accept;
  }
}
Conformité ISO 27001 A.13.1.1 : Contrôle des connexions réseau avec journalisation des tentatives bloquées. Le rate limiting sur SSH (10 connexions/minute) protège contre les scans automatisés.
5.2 Activation et vérification
# Activation du service :
sudo systemctl enable nftables --now

# Vérification des règles :
sudo nft list ruleset

6. Protection contre les attaques par force brute

Configuration Fail2ban

Détection et Réponse aux Incidents ISO 27001 A.16

Fail2ban implémente un système de détection d'intrusion basé sur les logs avec réponse automatique. Il protège contre les attaques par dictionnaire et force brute en analysant les tentatives d'authentification échouées et en bannissant temporairement les adresses IP suspectes.

6.1 Configuration de la prison SSH
# Fichier : /etc/fail2ban/jail.local

[sshd]
enabled = true
port = 2222
filter = sshd
logpath = /var/log/auth.log
maxretry = 3
findtime = 600
bantime = 3600
Conformité ISO 27001 A.16.1.2 : Surveillance proactive des événements de sécurité avec réponse automatisée. Paramètres calibrés (3 tentatives en 10 minutes, bannissement d'1 heure) pour équilibrer sécurité et disponibilité.
6.2 Activation du service
sudo systemctl enable --now fail2ban

6.2 Statut fail2ban

7. Gestion Avancée des Utilisateurs

Contrôle d'Accès Granulaire

Principe de Séparation des Tâches ISO 27001 A.9.2

La délégation granulaire des droits implémente le principe de séparation des tâches (SoD) et du moindre privilège. Chaque utilisateur reçoit uniquement les permissions nécessaires à ses fonctions, réduisant ainsi les risques d'erreur humaine et de malveillance interne.

7.1 Création de groupes de rôles
sudo groupadd admin
sudo groupadd user-mgmt
sudo groupadd pwd-reset
sudo groupadd auditor
7.2 Création d'utilisateurs et attribution
sudo adduser admin
sudo usermod -aG admin admin

sudo adduser nkuru
sudo usermod -aG user-mgmt nkuru

sudo adduser landry
sudo usermod -aG pwd-reset landry

sudo adduser yannick
sudo usermod -aG auditor yannick
7.3 Délégation granulaire des droits
# Fichier : /etc/sudoers.d/roles

%admin ALL=(ALL) ALL
%user-mgmt ALL=(ALL) /usr/sbin/adduser, /usr/sbin/userdel, /usr/sbin/usermod
%pwd-reset ALL=(ALL) /usr/bin/passwd
%auditor ALL=(ALL) /usr/bin/journalctl, /usr/bin/less /var/log/*

# Protection du fichier :
sudo chmod 440 /etc/sudoers.d/roles
Conformité ISO 27001 A.9.2.3 : Attribution des droits d'accès basée sur les rôles (RBAC). Les auditeurs peuvent consulter les logs sans les modifier, respectant ainsi l'intégrité des preuves numériques.
7.4 Politique d'expiration des mots de passe
sudo chage -M 90 -m 7 -W 14 admin
sudo chage -M 90 -m 7 -W 14 nkuru
sudo chage -M 90 -m 7 -W 14 landry
sudo chage -M 90 -m 7 -W 14 yannick

8. Gestion du Stockage avec LVM et Chiffrement

Stockage Sécurisé

Protection des Données au Repos RGPD Article 32

Le chiffrement LUKS2 avec LVM implémente une protection cryptographique des données au repos, exigée par le RGPD pour les données sensibles. LVM offre une flexibilité de gestion du stockage tandis que LUKS2 assure la confidentialité via l'algorithme AES-XTS (standard NIST).

8.1 Configuration LVM avec chiffrement
# Chiffrement avec LUKS2 :
cryptsetup luksFormat /dev/sdb1

# 1. Physical Volume :
sudo pvcreate /dev/mapper/crypt_data
# 2. Volume Group :
sudo vgcreate vg_storage /dev/mapper/crypt_data
# 3. Logical Volume :
sudo lvcreate -n lv_data -L 10 vg_storage
# 4. Formater :
sudo mkfs.ext4 /dev/vg_storage/lv_data
# 5. Vérification :
sudo pvs && sudo vgs && sudo lvs
8.2 Montage permanent
# 1. Obtenir l'UUID de la partition LUKS :
sudo blkid /dev/sdb1

# 2. Ajouter à /etc/crypttab :
echo "crypt_data UUID=xxxx-xxxx-xxxx none luks,discard" | sudo tee -a /etc/crypttab

# 3. Obtenir l'UUID du LV :
UUID_LV=$(sudo blkid -s UUID -o value /dev/vg_storage/lv_data)

# 4. Ajouter à /etc/fstab :
echo "UUID=$UUID_LV /mnt/data ext4 defaults 0 2" | sudo tee -a /etc/fstab

# 5. Tester :
sudo mount -a
df -h /mnt/data
Conformité RGPD Article 32 : "Pseudonymisation et chiffrement des données à caractère personnel". Le chiffrement au niveau bloc protège contre la perte/vol de matériel et l'accès physique non autorisé.

9. Protection du Chargeur de Démarrage

Sécurisation GRUB

Protection de la Chaîne de Démarrage ISO 27001 A.9.1

La protection de GRUB empêche les attaques au niveau du bootloader, un vecteur souvent négligé. Sans mot de passe, un attaquant pourrait démarrer en mode single-user ou modifier les paramètres du noyau pour contourner les contrôles de sécurité. Le hachage PBKDF2 assure une protection robuste contre les attaques par dictionnaire.

9.1 Génération du mot de passe haché
sudo grub-mkpasswd-pbkdf2
9.2 Configuration GRUB
# Fichier : /etc/grub.d/40_custom

set superusers="admin"
password_pbkdf2 admin <mot_de_passe_hashé>
# Mise à jour de GRUB :
sudo update-grub

12. Script d'Automatisation

Standardisation des Déploiements

Reproductibilité et Cohérence ITIL Service Design

L'automatisation via scripts assure la reproductibilité des configurations et élimine les erreurs humaines. Un script bien documenté permet un déploiement rapide et cohérent, essentiel pour la gestion de parc et la récupération après incident (DRP).

12.1 Script de configuration automatisée
#!/bin/bash
# setup-server.sh - Script d'automatisation du déploiement

apt update && apt upgrade -y
apt install -y nftables fail2ban lynis libpam-pwquality openssh-server rsyslog

sed -i 's/#PasswordAuthentication yes/PasswordAuthentication no/' /etc/ssh/sshd_config
systemctl restart ssh

# Pare-feu (exemple simplifié)
nft flush ruleset
nft add table inet filter
nft add chain inet filter input { type filter hook input priority 0 \; }
nft add rule inet filter input ct state established,related accept
nft add rule inet filter input tcp dport 2222 accept
nft add rule inet filter input drop
nft list ruleset > /etc/nftables.conf

echo "Configuration automatique terminée."
